## Context

当前 skill 库已经能递归读取 skill 目录，并在 `SkillSource.files` / `SkillSummary.filePaths` 中表示多文件结构。生成、合并和定稿保存流程也已经使用 `SkillFileDraft[]`，能够写入包含子目录的完整 skill。

缺口集中在现有 skill 详情编辑流：`SkillPreviewDialog` 只维护一个 `draftContent`，`/api/skills/update` 只接收 `content` 并写回 `SKILL.md`。因此复杂 skill 的附属文件只能被展示为路径列表，不能查看正文、新增、删除、编辑或随保存一起安全写回。现有 AI 生成和改写能力也以单个 `SKILL.md` 正文为中心，无法让 AI 基于聊天记录和用户提示生成或修改多文件 skill。

## Goals / Non-Goals

**Goals:**

- 在 skill 详情弹窗中显示完整文件树或文件列表，支持选择不同文本文件查看和编辑。
- 支持在同一编辑会话中新增和删除文本文件，并在保存时以完整草稿落盘。
- 为每个文件展示大小、可编辑状态和只读原因分类。
- 保存时提交完整文件草稿，确保 `SKILL.md` 和子目录文件作为同一次更新生效。
- 保留现有单文件 skill 的低成本体验：只有 `SKILL.md` 时仍表现为简单编辑器。
- 继续使用现有路径校验、目录名校验和原子替换能力，避免路径穿越和半写入状态。
- 让选区 AI 改写作用于当前正在编辑的文本文件，并在 UI 中明确当前文件路径。
- 让 AI 能基于聊天记录、当前复杂 skill 文件集、选中文件范围和提示词生成或修改多文件草稿。

**Non-Goals:**

- 不引入二进制资源编辑器、图片预览器或富文本编辑能力。
- 不支持在本次变更中重命名文件；用户可通过新增新路径并删除旧路径完成等价操作。
- 不改变 skill 存储位置或 `workspace/skills.available` / `workspace/skills` 的迁移规则。
- 不支持 AI 或手动编辑二进制文件内容；此类文件只能保留或删除。

## Decisions

- 使用带元数据的文件草稿作为编辑态数据模型，保存时再转换为 `SkillFileDraft[]`。
  - 理由：项目已经用该结构表示最终 skill 草稿，复用它能减少新模型和转换逻辑。
  - 具体字段包含相对路径、内容、大小、可编辑状态、只读原因、删除标记和是否新增。只读原因使用受控分类，例如 `binary`、`too-large`、`unsupported-encoding`、`protected`。
  - 替代方案：直接复用裸 `SkillFileDraft[]`。该方案无法承载文件大小、只读原因和删除状态，不满足新增需求。

- 扩展现有 `/api/skills/update` 支持 `files`，并在服务端通过完整草稿写回目标目录。
  - 理由：编辑已有 skill 与保存新 skill 的意图不同，继续使用 update 语义更清晰；服务端仍能复用 `validateFinalizedSkillDraft` 的路径和内容校验。
  - 替代方案：新增 `/api/skills/update-files`。该方案可降低兼容分支，但会增加一条只服务同一领域动作的 API。

- 保存复杂 skill 时采用完整目录替换，而不是逐文件覆盖。
  - 理由：完整替换能避免删除或遗漏旧子文件导致目录状态不一致，并可复用现有 staging/backup/rollback 思路。
  - 替代方案：逐文件写入。该方案实现更少，但失败恢复复杂，且容易留下旧文件。

- 新增文件和删除文件先只作用于客户端草稿，保存时统一提交完整文件集。
  - 理由：编辑会话可以支持撤销和重置，避免每次新增/删除都立即改变磁盘目录。
  - 替代方案：为新增/删除提供独立 API。该方案交互更即时，但会引入更多半完成状态和回滚路径。

- UI 将 `SKILL.md` 作为默认选中文件，并对当前选中文件复用 `SkillContentEditor`。
  - 理由：保持现有编辑器和选区改写能力，不引入新的大组件。
  - 替代方案：为多文件场景创建全新编辑器。该方案更灵活，但会放大改动范围。

- 仅允许编辑文本文件；无法安全解码为文本的文件显示为只读或不可编辑提示。
  - 理由：当前 API 和编辑器均以 UTF-8 文本为基础，二进制编辑会引入编码和文件损坏风险。
  - 替代方案：完全隐藏非文本文件。该方案会让用户误以为保存会丢失文件，不利于透明性。

- AI 多文件能力使用“完整草稿预览”作为输出边界。
  - 理由：生成复杂 skill、基于聊天记录更新复杂 skill、以及选区 + 提示词改写都可能影响多个文件；统一返回完整文件草稿便于用户逐文件审查后再保存。
  - 选区改写仍默认只替换当前文件的选区；如果用户明确要求跨文件调整，AI SHALL 返回多文件草稿预览而不是直接写盘。
  - 替代方案：让 AI 返回文本 diff。该方案节省 token，但前端应用补丁和冲突处理复杂度更高。

## Risks / Trade-offs

- [Risk] 完整目录替换可能影响目录更新时间或文件权限等元数据 → 使用现有写入策略并在文档中接受该限制，必要时后续再补充元数据保留。
- [Risk] 大型复杂 skill 的全部文件进入客户端状态可能带来性能压力 → 保持现有限长校验，优先支持常规文本 skill；超限时服务端返回明确错误。
- [Risk] AI 多文件输出可能误删或遗漏文件 → 保存前始终展示完整文件草稿，并要求 `SKILL.md` 存在；未保存前不写盘。
- [Risk] 新增/删除文件会扩大误操作风险 → 删除仅标记在草稿中，用户可重置；保存摘要中展示新增和删除数量。
- [Risk] 用户编辑多个文件后切换文件可能丢失选区改写上下文 → 文件切换时清理当前选区和改写预览，并在标题中显示当前文件。
- [Risk] 文件路径排序与目录层级展示可能影响可用性 → 初版使用稳定排序的路径列表，后续再增强为树形展示。

## Migration Plan

- 保持现有 `content` 请求兼容，或在前端同步迁移到 `files` 后保留服务端兼容分支以降低风险。
- 对已有单文件 skill 无需数据迁移；读取后会形成只包含 `SKILL.md` 的文件草稿。
- AI 接口先兼容旧单文本输入，再逐步增加多文件输入/输出字段；旧流程可继续生成单个 `SKILL.md`。
- 回滚时可恢复到只编辑 `SKILL.md` 的 UI 和 update 行为；底层递归读取与定稿保存逻辑无需回滚。

## Resolved Questions

- 同一版本需要支持新增/删除文件，但不支持重命名的专门交互。
- 文件列表需要展示文件大小和更细粒度的只读原因。
- 多文件场景需要保留并扩展现有 AI 生成、基于聊天记录更新、选区 + 提示词改写能力。
